# Task Manager Web Application (MVP)

## Проблема
Користувачі стикаються з проблемами організації завдань:
- відсутність чіткого контролю за виконанням;
- складність координації у командній роботі;
- існуючі сервіси занадто складні або перевантажені функціями.

---

## Пітч та Цільова аудиторія
**Пітч:** Web Task Manager — простий вебзастосунок для створення, редагування та контролю завдань у особистому або командному форматі.  
**Цільова аудиторія:** студенти, команди проєктів, користувачі, які хочуть ефективно організовувати робочі чи навчальні завдання.

---

## Мета (Goal)
Створити MVP, який дозволяє користувачам:
- створювати завдання;
- редагувати завдання;
- змінювати статус виконання;
- переглядати список завдань.

---

## MVP (Ключовий сценарій)
Користувач:
1. Створює нове завдання.
2. Завдання відображається у списку.
3. Користувач редагує завдання або змінює його статус.
4. Завдання позначається як виконане.

---

## Критерії успіху та приймання
- Завдання додається не більше ніж за 10 секунд.  
- Статус змінюється коректно.  
- Новий користувач може виконати сценарій без інструкцій.  
- Відсутні критичні помилки під час тестування.  
- Середній час виконання сценарію ≤ 20 секунд.  

---

## План реалізації (Завдання)
- [ ] Створення форми додавання завдань  
- [ ] Відображення списку завдань  
- [ ] Редагування завдань  
- [ ] Зміна статусу виконання (active / completed)  
- [ ] Тестування ключового сценарію  

---

## Стейкхолдери

**Кінцеві користувачі (студенти, команди проєктів):**  
Очікують простий та зрозумілий інтерфейс без перевантаження функціями. Важлива швидкість додавання завдань і можливість швидко змінювати статус.

**Власник продукту / Автор проєкту:**  
Зацікавлений у перевірці гіпотези, що простий інструмент буде зручнішим за перевантажені аналоги. Обмеження — час реалізації MVP та технічні ресурси.

**Викладач / Оцінювач:**  
Оцінює логіку формування вимог, відповідність MVP критеріям приймання та наявність вимірюваних показників успіху.

---

## Користувацький сценарій

**Умови початку:**  
Користувач відкриває вебзастосунок. Список завдань порожній або містить раніше створені записи.

**Кроки:**  
1. Користувач вводить назву завдання у форму.  
2. Натискає кнопку додавання.  
3. Завдання з’являється у списку.  
4. Користувач змінює статус завдання на виконане.  
5. За потреби редагує текст завдання.

**Очікуваний результат:**  
Завдання збережене у списку, його статус коректно оновлюється, зміни відображаються без перезавантаження сторінки.

---

## Перелік фіч першого інкременту

- Додавання нового завдання  
- Відображення списку завдань  
- Редагування тексту завдання  
- Зміна статусу (active / completed)  
- Мінімальна валідація введення (заборона порожнього завдання)

---

## План перевірки гіпотез

**Гіпотеза 1:** Користувачам потрібен максимально простий інтерфейс без зайвих функцій.  
Перевірка: спостереження за проходженням ключового сценарію без інструкцій.

**Гіпотеза 2:** Час виконання сценарію не перевищує 20 секунд.  
Перевірка: вимірювання часу під час тестування з 3–5 користувачами.

**Гіпотеза 3:** Основна цінність — швидке фіксування та завершення завдань.  
Перевірка: короткий зворотний зв’язок після першого використання.

---

## Основні невизначеності та способи їх зменшення

- Можлива перевантаженість інтерфейсу → обмеження функціоналу лише базовими операціями.
- Непередбачені технічні помилки → тестування ключового сценарію перед публікацією.
- Низька зрозумілість → перевірка на новому користувачі без пояснень.

---

## Definition of Success (MVP)

MVP вважається успішним, якщо:

- Користувач проходить повний ключовий сценарій без інструкцій.
- Середній час виконання сценарію ≤ 20 секунд.
- Відсутні критичні помилки під час тестування.
- Щонайменше 70% тестових користувачів оцінюють інтерфейс як “зрозумілий”.
